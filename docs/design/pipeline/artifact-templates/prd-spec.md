## 章节一览


| §   | Markdown 标题  | 必填  | 说明                                             |
| --- | ------------ | --- | ---------------------------------------------- |
| —   | `# {title}`  | 必填  | 任务标题                                           |
| 1   | `## 概述`      | 必填  | 1～3 句目标；Profile、Revision                       |
| 2   | `## 术语与领域概念` | 必填  | 无专用术语时写 **「无」**                                |
| 3   | `## 背景与上下文`  | 必填  | **项目背景**叙述：为何做、为谁、何种场景；宜段落/bullet，**一般不用表**    |
| 4   | `## 业务指标`    | 选填  | 默认保留章节：有 KPI 写表，**无则写「无」**；与 §1/§10 完全重复时可整节省略 |
| 5   | `## 功能`      | 必填  | 能力级清单（FEAT-*）                                  |
| 6   | `## 用户故事`    | 必填  | 场景级（US-*）；建议 ≥2 条                              |
| 7   | `## 需求池`     | 必填  | 可排期交付细项（REQ-*）；建议 ≥1 条                         |
| 8   | `## 范围`      | 必填  | 纳入 / 排除边界                                      |
| 9   | `## 非功能性需求`  | 必填  | 性能、稳定性、一致性等；子标题不涉及时可写 **「无」**                  |
| 10  | `## 验收标准`    | 必填  | 可执行门禁（AC-*）                                    |
| 11  | `## 约束`      | 必填  | 无则写「无额外约束」                                     |


**页脚元数据（必填，放文末）：**

```markdown
---
task_profile: {profile}
revision: {revision}
parent_task_id: {parent_task_id 或 —}
```

## 写作前：几个概念怎么用


| 概念       | 回答的问题                | 典型 ID  |
| -------- | -------------------- | ------ |
| **功能**   | 交付哪些 **能力**？         | FEAT-* |
| **用户故事** | **谁**怎么用、为何？         | US-*   |
| **需求池**  | 可排期 **交付细项**、优先级与前置？ | REQ-*  |
| **业务指标** | 业务上怎样算「做成了」？（可选）     | KPI-*  |
| **验收标准** | 怎样判定 **可交付**？        | AC-*   |


**层次关系（勿混写）：**

```text
FEAT-1 待办 CRUD
  ├── US-1 添加待办
  ├── US-2 重启后仍能看到列表
  └── REQ-2 CLI 子命令（关联 FEAT-1，前置 REQ-1）

FEAT-2 本地持久化
  ├── US-2（共用）
  └── REQ-1 持久化（关联 FEAT-2）
```

- **功能** = 模块级能力；**用户故事** = Connextra 用户句式；**需求池** = 实现/交付向细项（可写子命令、校验规则等）。
- **业务指标** 偏业务结果；**验收标准** 偏可执行检查。无 §4 时，以 §10 为准。
- **追溯：** P0 功能宜关联用户故事；用户故事行为宜被验收标准（或 §4 业务指标）覆盖；需求池「前置」表达交付顺序，不靠优先级硬排。

## 各节写法



### §1 概述

- 文首 `# {title}` 为任务标题，与 §1 所述任务一致；首段 1～2 句 **目标**，不展开功能列表（功能在 §5）。
- 列表：`Profile`、`Revision`；增量任务时注明父任务。



### §2 术语与领域概念

表格列：**名词 | 解释**

- 收录正文中会反复出现的 **领域名词** 及一句话定义。
- 每条一名一词一释；解释说明「是什么 / 不是什么」。
- 正文有未定义领域词时须收录；无专用术语时写 **「无」**。
- **与 §5 区别：** §2 讲名词含义；§5 讲交付能力。



### §3 背景与上下文

交代 **与项目相关的背景**，让读者（HITL、Architect）理解需求从哪来、在什么前提下成立。**宜用 1～3 段叙述或 bullet 列表**，把语境写清楚；**一般不用表格**（表格留给 §2 术语、§5 功能等结构化章节）。

**写什么（按需覆盖，自然写入正文即可）：**

- **动机 / 场景：** 为什么要做、解决什么问题、典型使用情境。
- **用户与交付形态：** 谁在用、通过什么方式接触产品（如本地 CLI、内部 Web、对外 API）。
- **环境前提：** 是否联网、是否持久化、是否多用户等 **与理解需求相关的** 事实（用自然语言说明，不必拆成「键 | 值」配置表）。

**与其它节分工：**


| 章节            | 区别                                               |
| ------------- | ------------------------------------------------ |
| **§1 概述**     | 1～2 句 **目标**；§3 **展开背景**，不重复堆砌功能列表               |
| **§5 功能**     | §3 讲 **语境**；§5 讲 **交付什么能力**                      |
| **§8 范围**     | §3 讲 **背景与前提**；§8 讲 **本次做 / 不做的边界清单**            |
| **§9 非功能性需求** | §3 用 **业务语言** 交代场景；§9 写 **运行态假设**（体量、并发、性能、一致性等） |


叙述示例见 [完整示例 · 背景与上下文](#完整示例)。

### §4 业务指标（可选）

**何时写表：** 有明确业务目标或结果度量（`KPI-`* + 表列）。

**默认：** 保留 `## 业务指标` 章节——有 KPI 写表，**无则写「无」**；验收以 §10 为准。仅当业务目标已在 §1 + §10 说清、且保留本章无增量信息时，可 **整节省略**。

推荐表列：`ID` | 指标名称 | 说明 | 目标值 | 验证方式

- ID 形如 `KPI-1`。
- 写 **业务结果**（如「用户能完成核心流程」），不写「测试通过」「lint 无报错」——那是 §10。
- `target` 可引用 `US-*` 或 `AC-*`；验证方式：`automated_test` / `manual` / `deploy_check` / `lint`。

**有 KPI 时表例（片段）：**

```markdown
## 业务指标

| ID | 指标名称 | 说明 | 目标值 | 验证方式 |
|----|----------|------|--------|----------|
| KPI-1 | 典型算式正确 | 四则、括号、小数组合表达式结果正确 | 见 AC-2 | manual |
```



### §5 功能

推荐表列：`ID` | 功能名 | 描述 | 优先级 | 关联用户故事

- ID 形如 `FEAT-1`；优先级 `P0` | `P1` | `P2`（P0 = 本次必须交付）。
- 写 **能力级**说明（如「四则运算」「本地持久化」），**不要**写 Connextra 用户句式。
- P0 功能宜在「关联用户故事」列挂至少 1 个 `US-*`。


|     | 示例                     | 评价          |
| --- | ---------------------- | ----------- |
| ✓   | FEAT-1 待办 CRUD（P0）     | 能力级         |
| ✗   | FEAT-1 As a 用户 I want… | 用户故事句式 → §6 |




### §6 用户故事

推荐表列：`ID` | As a / 作为 | Want / 我想要 | So that / 以便  
（或 Connextra 整句，中/英均可。）

**句式：**

```text
作为{角色}，我希望{可观察行为}，以便{业务价值}。
As a {role}, I want {observable behavior}, so that {value}.
```


| 部分     | 规则                                           |
| ------ | -------------------------------------------- |
| **角色** | 具体 Persona（如「CLI 使用者」）；优先用 §2 已定义角色；避免泛称「用户」 |
| **行为** | 用户 **可观察** 的结果；不写实现（如「用 JSON 存储」）            |
| **价值** | 一句业务动机                                       |


- ID 形如 `US-1`，全文唯一。建议 ≥2 条：主流程 + 边界/异常之一。
- **异常场景（建议）：** 有用户可感知的失败时，增加异常向 US；§10 须能验收。


|     | 写法                                                | 评价        |
| --- | ------------------------------------------------- | --------- |
| ✓   | 作为 **CLI 使用者**，我希望 **添加待办并能在列表中看到**，以便 **记录任务**   | 行为可见      |
| ✗   | 作为用户，我希望 **实现 add 子命令**                           | 实现细节 → §7 |
| ✓   | 作为 **CLI 使用者**，输入 `**2/0` 时我希望看到明确错误提示**，以便知道算式无效 | 异常 US     |
| ✗   | 作为用户，我希望 **抛出 ZeroDivisionError**                 | 实现 → §7   |




### §7 需求池

推荐表列：`ID` | 描述 | 优先级 | 关联功能 | 前置（可选）

- ID 形如 `REQ-1`；每条 **必须** 有优先级。
- 写 **可排期、可交付** 的细项（子命令、校验、存储格式等），可略偏实现。
- **前置** 列：交付级依赖（如 REQ-2 依赖 REQ-1）；无则 `—`。
- **不要** 与 §5 功能描述逐字相同；**不要** 把 §6 的 `want` 原样复制；**不要** 用 Connextra 写 REQ。


| 用户故事（§6）       | 需求池（§7）                       | 前置    |
| -------------- | ----------------------------- | ----- |
| US-1：添加待办      | REQ-2：实现 `add` 子命令与参数校验（P0）   | REQ-1 |
| US-2：重启后仍能看到列表 | REQ-1：todos 持久化到单 JSON 文件（P0） | —     |




### §8 范围

推荐用两个小标题 + bullet（或短列表），**不要用英文键名**（如 scope_in）：

**本次包含**

- 写 **交付边界**（如「本地 CLI」「含自动化测试」），粒度粗于 §5 功能表。
- 勿逐条复述 §5；勿与 §3 背景重复（§3 已写的语境不必再写）。

**明确不做**

- 写 **产品级排除项**（如「不做 Web UI」）。无则写「无」。

**与 §11 区别：** §8 **本次包含** = 交付边界；§8 **明确不做** = 产品级排除；§11 = **不可违反** 的硬规则（如禁止密钥入库），不是「本次不做」清单。

### §9 非功能性需求

写 **非功能性需求（NFR）** 基线：**中文说明为主**，必要时括号带短标签。**不要**写 p99、QPS、RPO/RTO 等具体数值。

推荐结构：

1. **一段总述**（可选，概括运行态假设）。
2. `**### 稳定性与性能**` — 用户体量、高并发、性能预期等。
3. `**### 数据一致性**` — 一致性模型、投递语义、多写者、幂等、冲突策略等。

**某块不涉及时：** 保留对应 `###` 小标题，正文写 **「无」** 即可（勿省略小标题导致章节结构不完整）。例如纯批处理、无共享状态的任务，「数据一致性」可写「无」。

有 NFR 内容时，各块用简表（项 | 说明）。稳定性与性能见 [完整示例](#完整示例)；涉数据一致性见 [补充片段](#补充片段)。

### §10 验收标准

推荐表列：`ID` | 描述 | 验证方式

- ID 形如 `AC-1`。
- 写 **可判定是否可交付** 的检查项；描述宜可执行。
- **验证方式**（择一）：`automated_test`（自动化测试）| `manual`（手动走查）| `deploy_check`（部署/环境检查）| `lint`（静态检查）。
- 与 §4 分工：§10 = 流水线 **门禁**；§4 = **业务结果**目标（若有）。勿把「测试通过」当业务指标写在 §4。
- 宜至少 1 条 `automated_test`；有异常向用户故事时须有 AC 覆盖（细则见 §6）。



### §11 约束

逐条列出 **硬规则**（非「本次不做」——后者见 §8 **明确不做**）。无则写 **「无额外约束」**。

常见示例：禁止密钥入库（`no_secrets_in_repo`）、禁止日志含 PII（`no_pii_in_logs`）。

## 完整示例

以下为 **典型小 CLI** 的 Run `prd.md` 样例：§4 无业务指标写「无」；§9「数据一致性」不涉及写「无」；用户故事用中文表头。

```markdown
# CLI 四则运算计算器

## 概述

命令行输入数学表达式，支持加减乘除、括号与小数，输出求值结果。

- **Profile：** default
- **Revision：** 1

## 术语与领域概念

| 名词 | 解释 |
|------|------|
| 表达式 | 用户输入的算式字符串，由数字、运算符、括号组成 |
| 运算符 | 本产品中指 +、-、*、/ 四则运算符号 |
| 小数 | 含小数点的数值字面量（如 3.14、.5） |
| CLI 使用者 | 在命令行输入表达式并查看结果的人（本示例唯一角色） |

## 背景与上下文

面向需要在终端快速验算的用户，以本地 CLI 交付：用户在本机命令行输入表达式即可得到结果，无需联网、不保存历史，单次运行、无多用户协作。

## 业务指标

无

## 功能

| ID | 功能名 | 描述 | 优先级 | 关联用户故事 |
|----|--------|------|--------|--------------|
| FEAT-1 | 四则运算 | 支持 +、-、*、/ 及运算符优先级 | P0 | US-1, US-4 |
| FEAT-2 | 括号求值 | 支持 () 改变运算顺序，含嵌套括号 | P0 | US-2 |
| FEAT-3 | 小数运算 | 支持小数输入与输出；除法产生小数结果 | P0 | US-3 |

## 用户故事

| ID | 作为 | 我想要 | 以便 |
|----|------|--------|------|
| US-1 | CLI 使用者 | 输入 1+2*3 等四则表达式并得到正确结果 | 快速完成基础计算 |
| US-2 | CLI 使用者 | 输入 (1+2)*3 等含括号的表达式并得到正确结果 | 计算需改变优先级的算式 |
| US-3 | CLI 使用者 | 输入 3.5/2 等含小数的表达式并得到正确结果 | 处理非整数计算 |
| US-4 | CLI 使用者 | 输入 2/0 或非法表达式时看到明确错误提示 | 知道输入有误而非程序崩溃 |

## 需求池

| ID | 描述 | 优先级 | 关联功能 | 前置 |
|----|------|--------|----------|------|
| REQ-1 | 表达式词法/语法解析（数字、运算符、括号、小数） | P0 | FEAT-1 | — |
| REQ-2 | 求值引擎：四则运算与运算符优先级 | P0 | FEAT-1 | REQ-1 |
| REQ-3 | 括号嵌套求值 | P0 | FEAT-2 | REQ-2 |
| REQ-4 | 小数解析与除法小数结果处理 | P0 | FEAT-3 | REQ-2 |
| REQ-5 | 除零与非法表达式错误提示 | P0 | FEAT-1 | REQ-2 |

## 范围

**本次包含**

- 本地 CLI 表达式求值（单次或 REPL）
- 含自动化测试与基本错误处理

**明确不做**

- Web / GUI
- 科学函数（sin、sqrt 等）、取模、幂运算
- 计算历史持久化

## 非功能性需求

本地 CLI、无状态单次求值，无持久化、无高并发。

### 稳定性与性能

| 项 | 说明 |
|----|------|
| 用户体量 | 个人使用 — 本地 CLI |
| 高并发 | 否 — 无并行请求场景 |
| 性能预期 | 尽力而为 — 交互可感知即可，不设 SLA |

### 数据一致性

无

## 验收标准

| ID | 描述 | 验证方式 |
|----|------|----------|
| AC-1 | 自动化测试套件全部通过 | automated_test |
| AC-2 | 手动走通 US-1～US-3 典型表达式，且 US-4（如 2/0）有明确错误提示 | manual |

## 约束

无额外约束

---

task_profile: default
revision: 1
parent_task_id: —
```



### 补充片段

**涉数据一致性时（§9 片段，有持久化 / 多写者等场景）：**

```markdown
### 数据一致性

| 项 | 说明 |
|----|------|
| 一致性模型 | 本地单写 — 单 JSON 文件为唯一数据源 |
| 投递语义 | 尽力而为 — 进程异常退出可能丢末次未落盘写入 |
| 多写者 | 否 |
| 须幂等 | 否 |
| 冲突策略 | 不适用 |
| 补充说明 | 单用户 CLI；无跨进程共享状态 |
```
