## 章节一览

### 正文


| §   | Markdown 标题        | 必填  | 说明                                                |
| --- | ------------------ | --- | ------------------------------------------------- |
| —   | `# 设计文档 — {title}` | 必填  | 建议含版本、对应需求、状态                                     |
| 1   | `## 1. 背景与上下文`     | 必填  | 本设计解决什么；参与者与边界（角色级）                               |
| 2   | `## 2. 设计目标`       | 必填  | 可验证目标；宜引用 spec 的 FEAT-* / US-* / REQ-*            |
| 3   | `## 3. 非目标`        | 必填  | 显式排除；与 spec 范围排除对齐                                |
| 4   | `## 4. 方案设计`       | 必填  | 本章必填；**4.1～4.7 子节按需选用**（见 [§4 子节](#4-方案设计--固定子节)） |
| 5   | `## 5. 非功能性目标`     | 选填  | 承接 spec §9 的 **设计侧量化**（QPS/TPS、延迟、可用性等）；小任务无增量时可整节省略          |
| 6   | `## 6. 测试用例列表`     | 必填  | `TC-*` 清单表；须含正向（HAP）+ 异常（NEG）+ 边界（BND）            |


### §4 方案设计 — 固定子节


| 子节  | 标题                 | 必填  | 说明                                                           |
| --- | ------------------ | --- | ------------------------------------------------------------ |
| 4.1 | `### 4.1 概述`       | 选填  | 架构风格 + 主数据/控制流；小任务可在 `## 4` 下一段总述代替                          |
| 4.2 | `### 4.2 系统架构图`    | 选填  | `architecture-*.mmd`；拓扑非一目了然或有外部中间件时写                        |
| 4.3 | `### 4.3 模块划分`     | 选填  | 多模块、域前缀、路径规划；单文件脚本可省略                                        |
| 4.4 | `### 4.4 外部依赖`     | 选填  | 中间件（DB/缓存/MQ 等）**或外部第三方系统**（支付、OAuth、SaaS API 等）；无则 **整节省略** |
| 4.5 | `### 4.5 接口定义`     | 选填  | 模块 API、入参出参、错误码；极简可在 §4.3 或附录 B 一带而过                         |
| 4.6 | `### 4.6 存储结构` | 选填  | 持久化 / 缓存 / MQ schema；大体量时 **`#### 数据量与扩展`**（预估、分库分表、读写分离）；节内 **`####` 分块**；无则 **整节省略** |
| 4.7 | `### 4.7 流程与时序`    | 选填  | 多分支 / 异步 / 需 US 图形追溯时写；**业务流 / 数据流 / 资金流均选填**（见 [§4.7](#47-流程与时序选填)）；线性小 CLI 可 **整节省略** |

## 写作前：与 spec 的分工


| 文档            | 回答                                     |
| ------------- | -------------------------------------- |
| **spec.md**   | 做什么、验收什么（[prd-spec.md](./prd-spec.md)） |
| **design.md** | 怎么做：模块、依赖、数据、接口、异常、测试、任务               |


**内容落点（无独立「横切」章）：**


| 议题              | 写在哪                                           | 何时写                                       |
| --------------- | --------------------------------------------- | ----------------------------------------- |
| 错误码 `ERR-*`     | **§4.5** 各操作「错误码」列（极简可写在 §4.3）；格式见 [标识符规范](#标识符规范错误码--测试用例) | 有用户可感知异常时（如 US-4 除零）                      |
| 事务 / 一致性 / 幂等   | **§4.6** 末段「一致性与事务」；或 **§4.7** 多步流程说明         | spec §9「数据一致性」非「无」、或多写者 / 跨模块写时           |
| 数据量预估 / 分库分表 / 读写分离 | **§4.6** `#### 数据量与扩展`（或表下简要一句） | spec §9 有 **体量/增长** 假设，或单表预估 **千万行 / 百 GB 级** 以上 |
| 业务流 / 数据流 / 资金流 | **§4.7**（**各自选填**；宜按 US 或流类型分 `####`） | **写了 §4.7 时**：主要 US 至少 1 图（多为业务流）；**数据流 / 资金流** 仅当跨存储同步或支付/结算等 **有设计增量** 时再画 |
| 微服务 **拆分边界与依据** | **§4.1**（一句）+ **§4.3 微服务一览** + **§3 非目标** | 单 spec 拆 N 服务时；原则见 [微服务拆分原则](#微服务拆分原则) |
| 非功能性目标（性能、可用性等） | **§5**                                        | spec §9 有 **设计侧增量** 时（数据一致性 → §4.6，不写 §5） |
| 鉴权 / 权限 / 安全策略   | **§4.4**（IdP/OAuth 等依赖）+ **§4.5**（鉴权失败 ERR）+ **§4.7**（登录/授权流程，选填） | spec 或 US 涉及登录、RBAC、敏感操作时 |
| 方案选型 / 为何拒绝备选   | **HITL / 评审记录**                               | 不在 `design.md` 重复                         |

## 各节写法

### §1 背景与上下文

- **1～3 句**说明本设计对应哪份 spec、解决什么问题。
- 可选 **参与者表**：名称 | 类型 | 说明（人、外部系统等，**角色级**）。
- **与 §4.4 分工（有外部依赖时）：** §1 写边界与角色；§4.4 写可连接、可版本化的依赖清单。无外部依赖时不写 §4.4。

### §2 设计目标 / §3 非目标

- **§2：** 可验证条目；引用 `FEAT-`*、`US-`*、`REQ-*`（及 spec 业务指标，若有）。
- **§3：** 与 spec **明确不做** 对齐；**不要**重复 spec 术语表或 §2 术语全文。

### §4 方案设计

`## 4. 方案设计` **必填**；其下 **4.1～4.7 均为选填子节**——只写对实现有增量的部分。

**选用原则：**


| 子节  | 何时写                                      | 不涉及时                           |
| --- | ---------------------------------------- | ------------------------------ |
| 4.1 | 需单独点明架构风格 / 主路径                          | 已在 §2、§4 首段说清则可省略              |
| 4.2 | 模块 ≥2、有中间件、拓扑不直观                         | **整节省略**                       |
| 4.3 | 多模块、域前缀、文件边界                             | 单文件脚本 **整节省略**                 |
| 4.4 | 有中间件、**外部第三方系统**或服务间调用                   | **整节省略**（勿写占位）                 |
| 4.5 | 需约定操作签名、错误码                              | 极简任务可并入 §4.3 或附录 B             |
| 4.6 | 有表/文件持久化、**Redis 等缓存键值结构**、MQ 消息体；大体量时含 **数据量与扩展** | 无跨请求结构化存储 **整节省略**（勿写「物理存储：无」） |
| 4.7 | 多分支、异步、**主要 US 需图形追溯** | 线性 pipeline **整节省略**（如计算器）；**业务流 / 数据流 / 资金流各自选填** |


**编号：** 已写子节须用固定标题 `### 4.x …`；**可跳号**（如仅有 4.1、4.3、4.5）。**禁止**用「无 / 不适用」占一节。

**§4 内部分块（`####`）：** **`### 4.1～4.7` 固定，不新增 `4.8+`**。某一节内容过多时，在 **该节内部** 用 `####` 拆分（只写有增量的块）。图表仍优先 **独立 `*.mmd`**，正文只引用文件名。

| `###` | 典型篇幅 | 是否宜 `####` 拆分 | 推荐 `####` 粒度 |
|-------|----------|-------------------|------------------|
| **4.1 概述** | 短（数段） | **一般不拆** | — |
| **4.2 架构图** | 短（图外置） | 多视角时可选 | `#### 逻辑视图` / `#### 部署视图`（各引用一张 `architecture-*.mmd`；**部署视图** = 组件/进程拓扑，**非** K8s/Rollout） |
| **4.3 模块划分** | 中～长 | **宜拆** | 多微服务：`#### 微服务一览` + `#### {服务} 内模块`；单体：`#### {域}模块` 或一张总表不拆 |
| **4.4 外部依赖** | 中 | 依赖种类 ≥2 时 **宜拆** | `#### 中间件` / `#### 第三方系统` / `#### 内部微服务` |
| **4.5 接口定义** | 长 | **必须拆** | **每模块或每微服务** 一个 `####`（计算器示例即此类） |
| **4.6 存储结构** | 长 | **必须拆** | 见下表 [§4.6 `####` 块](#46-存储结构选填) |
| **4.7 流程与时序** | 长 | **必须拆** | 内容长时 **须** 按 **US** 分 `####`；**业务流 / 数据流 / 资金流** 各 `####` **均选填** |

**何时不要继续拆：** 拆后每块仅 1～2 行 → 合并或整节省略该 `####`。**禁止**为凑结构开空 `####`。

### §4.1 概述（选填）

- 一句话架构风格（如 pipeline / layered）。
- 用 **1～2 句**说明主数据/控制流（如「CLI → Parser → Evaluator → stdout」）；**不要**在本节展开模块表或画图。

### §4.2 系统架构图（选填）

- **独立文件：** `architecture-*.mmd`，正文写文件名（必要时一句说明图意）。格式见 [flow-spec.md](./flow-spec.md)（`kind: context`）。
- **画什么：** §4.3 模块 + §4.4 外部依赖的 **静态拓扑**（谁连谁）；节点名与模块表一致。
- **何时写：** 模块 ≥2、有外部中间件、或拓扑非一目了然时 **建议写**；单模块小 CLI 等 **可整节省略**。
- **与 §4.7 分工：** 架构图 = 结构；时序/流程图 = 一次请求路径、分支与异常。
- **部署视图：** 若写 `#### 部署视图`，指 **逻辑组件/进程** 拓扑，**不含** K8s 清单、发布回滚（见文首范围说明）。

### §4.3 模块划分（选填）

推荐表列：**模块 | 路径 | 职责 | 域前缀 | 依赖模块**

- **内容多时的 `####`：** 微服务场景用 `#### 微服务一览` + 各服务 `#### {服务} 内模块`（见 [多微服务场景](#多微服务场景单-spec--单-designmd)）；模块 ≥10 可按域/包名分 `####`。
- **域前缀：** 大写缩写 2～12 字符，**每模块唯一**；用于 `ERR-{域}-`*、`TC-{种类}-{域}-`*（见 [标识符规范](#标识符规范错误码--测试用例)）。
- 访问 DB / Redis / HTTP 等宜经 **Repository / Client** 模块封装，业务模块不直连中间件；该中间件相关错误码 **优先用封装模块的域前缀**。

### §4.4 外部依赖（选填）

**有** 本系统需连接、但 **不由本需求实现** 的依赖时写本节，包括：


| 类别               | 示例                                                   |
| ---------------- | ---------------------------------------------------- |
| **中间件 / 基础设施**   | 数据库、缓存、消息队列、对象存储、文件系统                                |
| **外部第三方系统**      | 支付网关（Stripe/支付宝）、短信/邮件、OAuth/IdP、物流、地图、SaaS Open API |
| **内部微服务**（多服务场景） | 同需求内其他微服务（类型可写「内部微服务」）                               |


推荐表列：**名称 | 类型 | 技术/版本 | 用途 | 关键性**

- **类型** 示例：数据库、缓存、消息队列、文件系统、**第三方系统**、HTTP API、内部微服务。
- **与 §4.5 分工：** §4.4 = **我们依赖谁**（连接清单）；§4.5 = **我们对外提供什么**（本模块/服务 API）。第三方系统的 **调用契约** 可在 §4.5 的 Client 模块中展开，但 **依赖本身** 须在 §4.4 登记。
- **与 §1 分工：** §1 参与者表为 **角色级**（人、外部机构）；§4.4 为 **可连接、可版本化** 的技术依赖。
- **域前缀：** 由 §4.3 中封装该依赖的模块承担；仅当无封装模块、且需独立 `ERR-`* 时，在依赖说明中补充域前缀（与模块域不重复）。
- **无任何上述依赖：** **不要**出现 §4.4（边界已在 §1 / §3 说明即可）。
- **依赖条目多时的 `####`：** 按 **中间件 / 第三方系统 / 内部微服务** 分块，每块一张表；种类单一则 **不拆**。

### §4.5 接口定义（选填）

**每模块一小节**，写清 **操作、功能说明、入参、出参、可能错误码**。§4.5 描述 **可调用契约（签名级）**，**不限于 HTTP REST**——按模块 **对外边界** 选用下表格式。

| 边界类型 | 小节标题 `{协议}` | 表头 | 何时用 |
|----------|-------------------|------|--------|
| **HTTP / 对外 API** | `HTTP` | 操作 \| 功能 \| **HTTP** \| 错误码；分操作展开入参/出参 | Web、微服务 REST、Open API |
| **CLI** | `cli` | 操作 \| 功能 \| **入参** \| **出参** \| 错误码（位置用 `argv`/`flag`） | 命令行入口 |
| **进程内 / internal** | `internal` | 同上简化表，或 入参 \| 类型 \| 必传 \| 描述 | 库模块、同进程调用 |
| **gRPC / WebSocket / 消息** | `gRPC` / `WS` / `MQ` 等 | 操作 \| 功能 \| **方法/主题** \| 错误码；消息体字段另表 | 非 REST 协议；与 §4.6 消息 schema 分工 |

**HTTP / 对外模块** — 操作总览 + 分操作展开：

```markdown
#### {模块名}（`{文件}` · HTTP · {域前缀}）

| 操作 | 功能说明 | HTTP | 错误码 |
|------|----------|------|--------|

**`{operation}` — 入参**

| 名称 | 位置 | 类型 | 必传 | 描述 |
|------|------|------|------|------|

**`{operation}` — 出参**

| 名称 | 类型 | 描述 |
|------|------|------|
```

**CLI / internal** — 简化总览表（计算器 [完整示例](#完整示例) 即此类）：

```markdown
#### {模块名}（`{文件}` · internal · {域前缀}）

| 操作 | 功能说明 | 入参 | 出参 | 错误码 |
|------|----------|------|------|--------|
```

- **与 §4.3 分工：** §4.3 = 边界与职责；§4.5 = **可调用契约**（签名级）。
- **第三方 Client：** 调用外部系统的契约写在封装该依赖的 **Client 模块** `####` 下；依赖本身仍登记 **§4.4**。

**错误码（`ERR-{域}-{序号}`）怎么写：**


| 写法                    | 适用                             |
| --------------------- | ------------------------------ |
| **逐操作列在 §4.5 表内**（推荐） | 模块不多；读者按接口查错误（计算器即此类）          |
| **§4.5 末追加「错误码一览」小表** | 模块多、ERR 分散；单独小表列码、名称、场景、可重试、处理 |


- 须覆盖 spec 中 **用户可感知** 的异常（如除零 US）；与 **§6** `TC-NEG-`* 一一可对照。
- **不要**另开独立章节重复罗列；无异常场景则 §4.5 不写错误码列即可。

### §4.6 存储结构（选填）

**推荐在 `### 4.6` 内用 `####` 分块**（只写有的块，**禁止**「Redis：无」类占位）。**不要**为此新增 `### 4.8` / `4.9`——仍归 §4.6 管辖。

**推荐顺序（可省略任意块）：**

| `####` 小节 | 何时写 | 写什么 |
|-------------|--------|--------|
| **逻辑模型** | 有实体关系时 | 实体、关系、不变量（文字或简表） |
| **PostgreSQL — …** / **表 {name}** | 有关系库或文档持久化 | 字段 **类型、可空、键、注释**；宜含审计字段与 **索引** 说明 |
| **Redis — …** | 有自定义键值 / 缓存 schema | 键模式、Redis 类型、字段、TTL、与 DB 同步/失效 |
| **消息 — …** | 本需求 **生产或消费** 带 schema 的消息 | topic/队列、消息体字段、分区键、幂等键、重试/死信约定 |
| **数据量与扩展** | spec §9 有体量/增长假设，或 **单表预估达千万行 / 百 GB 级** | 表级数据量预估；**分库分表**、**读写分离**、归档/冷热分层（见下） |
| **一致性与事务** | 跨存储、多写者、须幂等/事务时 | **固定放最后一节**；DB 事务、缓存/MQ 与 DB 一致性、幂等 |

**与 §4.4 分工：** §4.4 = **连哪个** Redis/MQ/DB（清单）；§4.6 的 `#### Redis — …` / `#### 消息 — …` = **数据结构长什么样**。

- **Redis 透传缓存**（整段响应原样缓存、无固定键 schema）：可不单独开 `#### Redis`，或在逻辑模型一句带过。
- **MQ 仅基础设施、本需求不定义消息体：** 不写 `#### 消息`（§4.4 登记即可）。
- **无持久化 / 仅进程内类型：** **整节 §4.6 省略**（类型可在 §4.5 入参出参中体现）。

**`#### Redis — …` 表列（推荐）：** 键模式 | 类型 | 字段/值 | TTL | 说明

**`#### 消息 — …` 表列（推荐）：** topic/队列 | 方向（产/消） | 消息体字段 | 分区/顺序键 | 幂等键 | 说明

**`#### 数据量与扩展`（选填于 §4.6 内）：**

承接 spec §9 **用户体量 / 数据增长** 等假设，写 **设计侧量化** 的存储规划。**与 §5 分工：** §5 = **QPS/TPS、延迟** 等吞吐与时延；§4.6 本节 = **行数、容量、分片与读写的数据面规则**。

| 何时写 | 何时可省略 |
|--------|------------|
| spec §9 明示大体量、快速增长或长期留存 | 个人 CLI、单用户 JSON、预估 **全表 < 百万行** 且无增长压力 |
| 单表上线即 **千万行 / 百 GB 级**，或 3 年预估触顶 | 本期明确 **不分表**，可在对应 `#### PostgreSQL` 下 **一句**说明预估即可 |

**推荐内容：**

1. **表级数据量预估** — 表 | 上线规模（行数/容量） | 增量（日/月） | 留存策略 | 备注
2. **分库分表**（数据量巨大或增长触顶时）— 是否分片；**分片键**、分片数/路由规则、跨分片查询约束、全局 ID 策略
3. **读写分离**（读压力或主库瓶颈时）— 写走主库；哪些读走从库/只读实例；延迟容忍与 **强一致读** 例外（须走主库的场景）
4. **归档 / 冷热分层**（可选）— 历史数据迁出、分区表、对象存储

**与 §4.4 分工：** §4.4 登记 **只读实例 / 分片中间件** 等依赖；本节写 **路由规则与分片策略**。

**本期不做分库分表时：** 仍宜写 **数据量预估** + **触发条件**（如「orders 超 5 千万行再按 user_id 水平分表」），避免设计留白。

### §4.7 流程与时序（选填）

**何时写整节：** 多模块协作、异步、跨服务、或 **主要 US** 主路径/异常 **非一眼可读** 时写；线性小 CLI（如计算器）**整节省略**。

**主要 US：** spec 中 **关联 P0 FEAT** 的 **`US-*`**（与 [prd-spec](./prd-spec.md) 一致；US 本身无 P0/P1 标签，以 FEAT 优先级为准）。

**主要 US（写了 §4.7 时须配图）：** 上列每个主要 US **至少 1 张图**（时序或流程均可，**通常即业务流**）；**不要求** 每个 US 再单独画数据流 / 资金流。

**推荐正文结构（写了 §4.7 时）：**

1. **US → 图表对照表**（写了 §4.7 时 **本节内必填**）— 列：**US | 图表文件 | 图类型 | 说明**；不得遗漏主要 US。图类型可写 `时序`、`流程`，或 `时序 + 流程`（**同文件多段** 或 **一张综合图** 均可，仍计为 **该 US 至少 1 张图**）。
2. **图表文件** — 独立 `flow-*.mmd` / `sequence-*.mmd`；正文写文件名，必要时一句图意。Mermaid 格式与 `diagrams[]` 登记见 [flow-spec.md](./flow-spec.md)。
3. **文字补充** — 正常路径 + 关键异常（与 §4.5 错误码、§6 用例一致）。

| 图类型 | 适用 |
|--------|------|
| **时序图** | 多参与方、跨模块/跨服务调用顺序 |
| **流程图** | 分支、重试、**异常路径**、补偿 |

**其它 US（P1/P2 或未挂 P0 FEAT）：** 可不单独配图，除非实现路径与主要 US 显著不同。

**命名：** 图中 participant / 节点与 §4.3 模块（或微服务）名一致。

**与 §4.2 分工：** 架构图 = 静态拓扑；§4.7 = **按 US 或流类型的一次动态路径**（含时序与异常）。

**业务流 / 数据流 / 资金流 — 均在 §4.7 内、各自选填（不另开 §4.8）：**

| 流类型 | 必填？ | 何时写 | 画什么 | 推荐图 |
|--------|--------|--------|--------|--------|
| **业务流** | **写了 §4.7 → 主要 US 宜有** | 完成 US 的 **步骤、分支、异常** 非一眼可读 | 主路径 + 关键异常/补偿 | **流程图**（主）；复杂协作加时序图 |
| **数据流** | **选填** | 跨 DB/Redis/MQ **同步路径复杂**、或 spec 要求图形化数据落点 | 数据/消息 **从哪产生 → 经谁 → 落哪** | **时序图**（主） |
| **资金流** | **选填** | 涉及 **支付、结算、退款、分账、对账** 等 | 钱从哪到哪、回调与不可逆点 | **流程图 + 时序图**（回调链） |

**分工：** 契约 → **§4.5**；存什么 → **§4.6**；§4.7 只画 **怎么流**（业务/数据/资金 **按需选用**，**禁止**为凑三种流各画一张空图）。

**写法建议：** 对照表可增加 **流类型** 列（业务 / 数据 / 资金），或按需开 `#### 业务流 — 下单` 等；**无支付则不写** `#### 资金流`；**无跨存储同步则不写** `#### 数据流`。

**§4.1** 仅用 1～2 句点明主业务流/主数据流，**不**替代 §4.7 图。

### §5 非功能性目标（选填）

承接 spec **§9 非功能性需求**，写 **设计侧可测、可验收** 的量化目标。**数据一致性 / 事务** 已在 **§4.6**，不在本节重复。

**与 spec 分工：** spec §9 写 **档位与假设**（如「高并发」「万级用户」「尽力而为」），**不写** p99、QPS、RPO 等具体数；**§5 写具体数值与拆分**（若设计侧有增量）。

**何时省略：** 小需求（本地 CLI、spec §9 为「尽力而为」且设计无额外容量/性能假设）——**整节省略**，勿复述 spec 原文。与 spec §9 相比 **无增量信息** 时，省略本章。

**何时写 QPS / TPS：**

| 情况 | 是否写 | 说明 |
|------|--------|------|
| spec §9 有性能/并发/体量假设 | **宜写** | 将定性档位 **量化** 为可验收目标 |
| 多微服务 / 网关 + 后端 | **宜写** | 可拆 **入口 QPS**、各服务 **TPS/QPS**、读/写分开 |
| 纯批处理、离线任务 | 用 **吞吐** 表述 | 如「批处理 TPS」「每日处理量」，不必强行写 QPS |
| spec §9「尽力而为」、无 SLA | **不写 §5** | 计算器级任务即此类 |

**QPS 与 TPS：** **QPS**（Queries Per Second）多用于 **读/query、GET、查询类** 接口；**TPS**（Transactions Per Second）多用于 **写操作、业务事务、下单/支付** 等端到端事务。同一系统可同时列两者（如「查单 QPS 5k，下单 TPS 500」）；团队统一用一种口径亦可，须在表内 **注明指标含义**（如「峰值 QPS（读）」「峰值 TPS（写）」）。

需写时推荐表列：**ID | 指标 | 目标 | 范围/备注**（如 `NFR-1`）

| 指标类型 | 示例目标列 |
|----------|------------|
| 吞吐 | 入口 QPS ≥ 3k；createOrder TPS ≥ 500 |
| 延迟 | P99 延迟 ≤ 200ms（读）；P99 ≤ 800ms（写） |
| 可用性 | 核心链路可用性 ≥ 99.9% |
| 容量 | 单实例支撑 500 TPS；水平扩展至 5 实例 |

- **多微服务：** 在「范围/备注」列标注 **责任服务或链路**（如「order-api 写路径」「经 API 网关入口」）。
- **须可验收：** 目标应能对应压测、监控或 AC/deploy_check（具体工具由 Profile / CI 定，不在此展开）。

## 标识符规范（错误码 & 测试用例）

> **说明：** 本节为 **规范文档** 作者参考；Run design.md **不设**独立「标识符规范」章，规则落在 §4.3 / §4.5 / §6。  
> **格式与正则：** [artifact-schemas/design-spec.md §标识符约定](../artifact-schemas/design-spec.md#标识符约定)（DES-023 / DES-024 校验基线）。

**原则：** `ERR-`* 与 `TC-`* **共用域前缀注册表**（以 §4.3 模块域为主；§4.4 仅在无封装模块时补充）。

### 格式一览


| 类型   | 格式                 | 说明                               |
| ---- | ------------------ | -------------------------------- |
| 错误码  | `ERR-{域}-{序号}`     | `{域}`=责任方；`{序号}`=该域内三位数字，从 001 起 |
| 测试用例 | `TC-{种类}-{域}-{序号}` | `{种类}`=HAP / NEG / BND           |


**种类：**


| 标记    | 含义                  |
| ----- | ------------------- |
| `HAP` | 正向 — 主路径成功          |
| `NEG` | 异常 — 失败 / 非法 / 依赖故障 |
| `BND` | 边界 — 空值、极值、深层嵌套等    |


**域前缀规则：** 大写字母开头，2～12 字符；每模块/外部服务 **一个域**；**谁抛出/谁被测用谁的域**。

**错误码表须写清：** 中文短名、触发场景、用户可见提示。

**联动：** 异常用例 `TC-NEG-{域}-`* 宜对应同域 `ERR-{域}-`*；正向用例不填错误码（格式见下方完整示例 · §6）。

**其它 id（可选）：** 非功能性目标 `NFR-{序号}` · 开发任务 `T{序号}`

## 多微服务场景（单 spec · 单 design.md）

**原则：** **一个需求（一份 spec）→ 一份 design.md`**。拆成 N 个微服务时，**N 个服务的边界、协作与数据归属都写进同一份设计**，**不要**按微服务拆成 N 份 design.md。

```text
docs/runs/<task_id>/
  spec.md          ← 一份需求
  design.md        ← 一份设计（内含 N 个微服务）
  architecture-*.mmd / flow-*.mmd
```

**何时仍用多份 spec / design：** 各服务 **独立需求、独立 AC、独立排期** 时，拆成多个 `task_id`（`parent_task_id` 指向父需求），每份 **1 spec + 1 design**。这与「单需求拆 N 服务、单 design」不矛盾——按 **需求边界** 拆文档，不按「每个服务一份 design」机械拆分。

### 微服务拆分原则

本节回答：**一次需求要拆成几个微服务、边界画在哪**。结论须落在 **§4.1 概述**（一句话说明拆分依据）、**§4.3 微服务一览**（表列职责与 **自有数据**）、**§3 非目标**（本次不拆出去的能力）。**方案对比与拒绝备选** 放 HITL / 评审记录，不在 design 重复。

#### 优先按什么拆（推荐）

拆分 **优先遵循 DDD 限界上下文与数据归属**；独立部署、变更频率等为工程层补充条件。

| 原则 | 说明 | design 里怎么体现 |
|------|------|-------------------|
| **领域 / 业务能力（Bounded Context）** | 一个微服务 ≈ 一个 **可独立讲清的业务能力**（订单、库存、支付、用户…） | §4.3 **职责** 列；附录 A 中 FEAT/US 落点 |
| **数据归属（谁写谁拥有）** | 每种 **主数据** 只有一个 **写方**；其它服务 **只读 API 或事件副本** | §4.3 **自有数据** 列；§4.6 **禁止多服务共表**（已在下文） |
| **独立部署与扩缩** | 流量、发布节奏、资源需求 **显著不同** 的能力宜分开 | §5 QPS/TPS 按服务拆；§4.3 路径/仓库 |
| **变更频率** | 常一起改、一起发布的放同一服务； **独立演进** 的拆开 | §4.3 依赖微服务；附录 C 任务按服务分组 |
| **事务与一致性边界** | **强一致、短事务** 尽量在同一服务 + 同一 DB；跨服务用 **API + 事件 + Saga/补偿** | §4.6 **一致性与事务**；§4.7 异常/补偿流程 |

#### 什么时候 **该** 拆

- spec §8 / §9 已假设 **多团队、高并发、独立扩缩**，或 P0 US **天然跨不同主数据**（如「下单」同时写订单与扣库存）。
- 某一能力 **技术栈或合规** 必须隔离（如支付 PCI 区、AI 推理 GPU 池）。
- 只读查询与写路径 **资源差一个数量级**，且缓存/读库策略不同。

#### 什么时候 **先不拆**（或保持模块化单体）

- 本次 run **无持久化** 或 **单库单服务** 即可满足 spec §9（见 [小任务极简指引](#小任务--无持久化极简指引)）。
- 拆开后 **每次主路径都跨 3+ 服务同步调用**，且无明确 Saga/事件设计 → **过度拆分**。
- 团队/发布 **仍是一体**，且无独立扩缩诉求 → 优先 **单服务内模块**（§4.3 模块表），不必为拆而拆。
- **强一致事务** 跨多写方且无法改为最终一致 → 优先合并到同一服务或重新划界。

#### 禁止 / 慎用的拆法

| 做法 | 问题 |
|------|------|
| **按技术层拆**（纯 API 层、纯 DAO 层、纯「微服务网关业务」） | 改一个 US 仍要改多服务，无领域内聚 |
| **按表拆**（每表一服务） | 忽视业务流程与事务边界 |
| **机械「一个 FEAT 一个服务」** | FEAT 粒度不等于部署边界 |
| **共享 DB 多服务写同一表** | 耦合隐藏、事务与归属不清（除非 §4.6 明确只读副本 + 写方唯一） |

#### 协作方式（拆完之后怎么连）

| 场景 | 推荐 | 写在哪 |
|------|------|--------|
| 需要立即知道结果（如库存够不够） | **同步 API**（调用方处理超时/降级） | §4.4 内部微服务 + §4.5 |
| 可最终一致（如通知、搜索索引、报表） | **异步消息 / 事件** | §4.6 `#### 消息`；路径复杂时 **可选** §4.7 数据流 |
| 跨服务失败需回滚 | **Saga / 补偿**（显式步骤，非隐式分布式事务） | §4.7 流程图异常分支 + §4.6 幂等 |
| 读多写少、读模型不同 | **CQRS / 读副本 / 缓存** | §4.6 Redis + §5 读 QPS |

#### 自检清单（写 §4.3 微服务一览前）

1. 每个服务的 **职责** 能否用 **一句动词短语** 说清？
2. 每张 **核心表 / 主数据** 是否 **唯一写服务** 明确？
3. 服务间调用是 **必要** 的，还是可合并为一个服务内的模块调用？
4. 每个 **主要 US** 的 **跨服务路径**（若存在）是否在 §4.7 有图、§6 有 **集成 TC**？
5. §3 **非目标** 是否列出 **本次不实现、也不预留** 的服务？

**示例（电商）：** `order-api`（订单写 + 查）与 `inventory-api`（库存写）拆分——因 **主数据不同**、**扩缩独立**、下单路径需 **同步预占** 但支付回调可 **异步**；**不** 单独拆「order-query-service」除非 spec §9 读 QPS 与写路径资源差显著且 §5 有量化依据。
