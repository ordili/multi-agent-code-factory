Redis 与 MySQL 一致性 — 最终架构方案

场景：高并发写入 / 秒杀 目标 TPS：10 万（同步写，不走异步落库） 存储：MySQL 分库分表 + Redis 缓存 版本：定稿 v1.5

一句话方案： 先写 MySQL（事务 + 版本号），再 CAS 更新 Redis（new_version > current 才写入）； 读路径命中零 DB；写路径与读回填均做 version 比较；Redis 更新失败重试后 DEL 或返回 503。

1核心原则（三条铁律）

01

MySQL 是唯一真相源（Source of Truth）— 对账、恢复、审计均以 MySQL 为准

02

写路径：先写 MySQL（事务提交成功）→ 再处理 Redis，不采用「先写 Redis」

03

缓存策略：version CAS（写路径 + 写回路径）；常规不 DEL；仅 Redis 更新失败时允许 DEL

2写路径（同步）

① 幂等校验（request_id + DB 唯一索引）
   → 重复请求直接返回，不重复 version++，不重复写 Redis
② 按业务键路由到同一 MySQL 分片（与 Redis hash tag 对齐）
③ 可选：热点 key 加短锁（Redis 分布式锁 / DB 行锁）
④ MySQL 本地事务写入（UPDATE ... WHERE + version++）
   → version 仅在「实际发生状态变更」时递增
⑤ 事务 commit 成功
⑥ Lua CAS 更新 Redis（new_version > current_version 才 SET）
⑥b CAS 返回拒绝（0）时：再读 DB 最新 (remain, version) → 用 DB 值再 CAS；仍失败 → DEL / 告警
⑦ Redis 更新失败处理：
   → 重试 2–3 次
   → 仍失败：DEL key（仅失败时允许删）或返回 503
   → 禁止静默降级（否则命中零 DB 会导致长期脏读）
⑧ 失效本节点 Caffeine 本地缓存（及可选 Pub/Sub 广播）
⑨ 返回成功

写路径 Redis 更新：Lua CAS（必须）

v1.2 修复：无条件 SET 会被慢写覆盖新值。写路径与读回填路径均须 version 比较。

-- 写路径 / 读回填 共用同一 Lua 逻辑
local cur = tonumber(redis.call('HGET', KEYS[1], 'version') or '0')
local new_ver = tonumber(ARGV[1])
if new_ver > cur then
    redis.call('HSET', KEYS[1], 'remain', ARGV[2], 'version', ARGV[1], 'expireAt', ARGV[3])
    return 1   -- 写入成功
end
return 0       -- 旧版本，拒绝覆盖

写路径并发覆盖示例（为何必须 CAS）

T1  写请求A：commit DB version=6，尚未更新 Redis
T2  写请求B：commit DB version=7，CAS Redis version=7 ✅
T3  写请求A：无条件 SET version=6 ❌ → 把新值覆盖成旧值
    CAS 方案：T3 时 new_ver(6) < cur(7) → 拒绝写入 ✅

Redis 更新失败处理（禁止静默降级）

步骤动作
1立即重试 Lua CAS，最多 2–3 次（指数退避）
2仍失败 → DEL key（仅失败时例外允许删，迫使后续读回源）
3或返回 503，由客户端重试（DB 幂等兜底）
❌ 禁止DB 已提交 + Redis 失败却返回成功且不做任何处理

明确不采用的方案

方案态度原因
先写 Redis，后写 MySQL 不采用 MySQL 失败需回滚 Redis，10W TPS 下回滚风暴与竞态难处理
写 MySQL 前删 Redis（写前删） 不采用 写未完成即掏空缓存，引发缓存击穿，DB 压力骤增
延时双删（DEL → 写 DB → DEL → sleep → DEL） 不采用 多次击穿窗口 + 写路径 sleep，10W TPS 不可接受
2PC / XA 跨 Redis + MySQL 不采用 延迟与锁持有时间过长，无法满足 10ms 级响应预算
异步 MQ 落库 不采用 本次约束要求同步写路径

3读路径

关键澄清：version CAS 写路径与读回填路径均须执行，不仅读回填。 命中且未逻辑过期 → 直接返回 Redis，零 DB 调用。 禁止「Redis 更新失败静默降级」— 与命中零 DB 矛盾。

version 校验发生在何时？

时机是否查 DB做什么
缓存命中（正常读） 不查 DB 直接返回 Redis 中的 value，Redis 价值在此体现
缓存 miss / 逻辑过期刷新 查 DB 一次 回源后 Lua CAS 写回；拒绝则再 GET Redis 返回新值
写路径推送缓存 先写 DB commit 后 Lua CAS 更新 Redis（new_version > current 才写入）
读回填写缓存 查 DB 一次 写回前 Lua CAS：new_version > current 才 SET；否则丢弃并再读 Redis
写后立刻读（强一致） 直读主库 绕过 Redis，不经过缓存层

路径 A：缓存命中 — 零 DB 调用（主路径，占 99% 流量）

GET Redis → { value, version, expireAt }

若 version = -1（失效标记）：
  → 强制走路径 B 回源

若 未逻辑过期（expireAt > now）：
  → 直接返回 value（零 DB 调用）✅

若 已逻辑过期：
  → 商品详情类：返回旧 value + 异步刷新
  → 库存数量类：走路径 B 同步刷新（避免展示有货实际售罄）

路径 B：miss / 过期刷新 — 仅一次 DB + CAS 写回

缓存 miss 或库存类逻辑过期
  → 互斥锁（Singleflight，同一 key 仅一个线程回源）
  → 读 MySQL 主库 → { value, version: 6 }
  → Lua CAS 写回 Redis（new_version > current_version）
  → 若 CAS 拒绝（已有更新版本）：
       丢弃回填 → 再 GET Redis 一次 → 返回 Redis 中的新值
  → 释放锁，其余等待线程读 Redis 返回

路径 C：写后强一致读 — 绕过 Redis

写后立刻读（下单结果、支付状态等）
  → 直接读 MySQL 主库
  → 不读 Redis（此类场景本就不需要缓存）

version 防旧值写回（无需额外查 DB）

时间线：
T1  线程A：写 DB version=6 → Lua CAS Redis version=6
T2  线程B：缓存 miss → 读 DB（从库延迟）→ 拿到 version=5
T3  线程B：准备 Lua CAS Redis version=5
    → 比较：db.version(5) < cache.version(6) → 拒绝写回 ✅
    （只读 Redis 中已有 version，不需再查 DB）

按数据类型区分读策略

数据类型逻辑过期策略原因
商品详情、图片、描述 返回旧值 + 异步刷新 短暂 stale 可接受，避免击穿
库存数量 同步刷新或直读 DB 避免页面显示有货、实际已售罄
写后立刻读（订单/支付） 绕过 Redis，直读主库 强一致要求

何时不需要 Redis？

场景建议
写后立刻读直读 MySQL 主库，绕过 Redis
QPS 低、强一致要求高直读 DB，可不用缓存
管理后台 / 对账直读 DB
10W TPS 热点读必须用 Redis，但命中时零 DB 调用

缓存击穿防护

手段说明
互斥回填（Singleflight） 热点 key 失效时，仅一个线程回源 DB，其余等待回填结果
逻辑过期（不物理 DEL） 非库存类：含 expireAt + version，过期后异步刷新；库存类不走此策略（见 §3 数据类型表）
版本号 CAS（写 + 写回） 写路径与读回填均 Lua CAS，防止新旧写并发覆盖
热点本地缓存（Caffeine） 极热 key 挡在 Redis 前；写成功后须失效本地缓存（本节点 + Pub/Sub 广播）

5一致性保障体系

手段作用层级
MySQL 事务 + 乐观锁 UPDATE ... WHERE remain > 0 + version++，防超卖 单分片强一致
幂等键 + 唯一索引 重复请求直接返回，不重复 version++，不产生副作用 接口层
版本号 CAS（Lua） 写路径推送 + 读回填均须 new_version > current；命中时不查 DB 缓存层
Redis 失败处理 重试 → DEL key 或 503；禁止静默降级 写路径
Caffeine 失效 写成功后失效本节点缓存；可选 Redis Pub/Sub 广播各节点 应用层
互斥回填 解决删缓存 / 失效带来的缓存击穿 读路径
写后读走主库 避免主从延迟导致脏读 读路径
定时对账（离线） 增量圈定 SKU → 读 summary 比对 Redis → CAS 修 Redis → 日志 + 告警 运维层
stock_summary 同事务更新 写路径与 segment 扣减同事务维护，对账免 SUM 大表 写路径

一致性级别定义

范围级别说明
单 MySQL 分片内 强一致 本地事务保证
Redis 缓存 最终一致 写路径 CAS 推送；毫秒~秒级 stale 可接受（库存读除外）
Redis + MySQL 跨存储 可收敛一致 非原子双写，靠 version + 对账收敛

6秒杀 / 热点 SKU 专项

即使 MySQL 总分片可扛 10W TPS，单个 SKU 仍可能是热点（单行库存锁竞争），需额外手段：

手段说明
库存分段（MySQL + Redis 对齐） DB 分段表 stock_segment(sku, seg_no, remain)；Redis 对应 {sku:1001}:stock:0~N；读时聚合 SUM，写时扣减对应分段并 CAS 更新该分段 key + 汇总 key
入口削峰 令牌桶 / 排队页，将流量削至可承受 QPS
热点本地缓存（Caffeine） 挡在 Redis 前；写成功后 必须失效（本节点 + Pub/Sub），或设极短 TTL（如 100ms）
库存读策略 库存数量 不用「逻辑过期返回旧值」，须同步刷新或直读 DB

库存扣减在 MySQL 分片内同步完成；Redis 承担 读加速 + 写后推送，真相源仍在 MySQL。

7分库分表与 Redis 路由对齐

shard_id = hash(business_key) % N

MySQL：写请求只打一个分片，禁止跨分片事务
Redis：同一 business_key 使用 hash tag，如 {sku:1001}:stock
路由键：user_id / sku_id / order_id（读写一致）

10W TPS 延迟预算（粗算）

步骤预算备注
MySQL 事务3 – 5 ms非热点 key
Redis Lua CAS1 – 2 ms—
网络 + 序列化2 – 3 ms—
合计≤ 10 ms仅适用于非热点 key
热点 SKU 单行单行锁瓶颈仍在 — 须库存分段 + 削峰，不能仅靠分片总数扛 10W

8Redis 缓存结构（推荐）

方案 A：单 key（非热点 / 低 QPS SKU）

Key:   {sku:1001}:stock          # hash tag 与分片对齐
Type:  Hash（便于 Lua CAS 单字段更新）
Fields: remain | version | expireAt
TTL:   物理 TTL 宜长于逻辑过期（如 24h）

方案 B：分段对齐（热点 SKU，与 §6 一致）

分段 key：{sku:1001}:stock:0  …  {sku:1001}:stock:N
汇总 key：{sku:1001}:stock:total   # remain = SUM(各分段)，读列表页用

写路径：
  ① 扣减某分段 DB 行 → commit
  ② CAS 更新对应分段 Redis key
  ③ CAS 更新汇总 key（remain -= 1, version++）

读路径：
  详情页库存 → 读 total key
  秒杀扣减   → 随机或轮询选分段

9讨论过程方案对比（最终取舍）

方案最终态度说明
先 Redis 后 MySQL 不采用 回滚难，10W TPS 风险大
异步 MQ 落库 不采用 本次约束：同步写
写后单删 Redis 仅低 QPS 热点场景引发缓存击穿
延时双删 不采用 多次击穿 + 写路径延迟
写 DB + Lua CAS 更新 Redis 最终写路径 写/写回均 CAS；失败重试后 DEL 或 503
互斥回填 + 逻辑过期 最终读路径 防击穿，热点友好

10定时对账（离线）

定位：在线一致性（幂等 + DB 事务 + Redis CAS）是第一道防线； 离线对账是兜底 — 以 MySQL 为准修复 Redis，发现偏差、自动收敛、超阈值告警。 默认 不反向修 DB（DB 异常须人工介入）。

10.1 对账解决什么？

在线机制盖不住的场景举例
Redis CAS / 更新失败后仅 DEL后续回填错误，remain 偏差
分段库存汇总 key 未同步total 与分段之和不一致
运维 / 脚本直接改 DBRedis 完全不知情
长期未访问的脏缓存无 miss，不回源，一直脏读

层级MySQL 真相Redis 现状优先级

10.5 依赖表结构（写路径须同事务维护）

表用途维护时机
stock_segment 分段库存明细 写路径扣减时更新
stock_summary 汇总 remain + version + updated_at 与 segment 扣减同一事务（对账主读表）

reconciliation_log 对账结果审计 对账任务写入

10.6 单 SKU 对账步骤

① 读 MySQL stock_summary（total_remain, version, updated_at）
② 读 Redis 汇总 key（方案 A：{sku}:stock；方案 B：{sku}:stock:total）
③ 比对：
   - redis miss          → CAS 回填，记 FIXED_MISS
   - remain 一致且 version 合理 → PASS
   - updated_at 在 5s 内  → SKIP_RECENT_WRITE（避免覆盖在线写入）
   - remain 不一致       → 进入修复
   - db_remain < 0       → 告警，人工介入，不自动修
④ Lua CAS 修复 Redis（与写路径同一套 CAS 逻辑）
⑤ 写 reconciliation_log；偏差 > 阈值则告警
⑥ 可选：抽样比对各分段 stock:0~N

10.7 对账修复 Lua

注意：FORCE_FIXED 仅用于 remain 不一致且 Redis version 虚高（脏数据）的修复；version 取 max(db_ver, cur_ver)，禁止回退 version。

-- 以 DB 为准，但不用旧 version 覆盖新写入
local cur_ver = tonumber(redis.call('HGET', KEYS[1], 'version') or '0')
local db_ver  = tonumber(ARGV[1])
local db_remain = ARGV[2]

if db_ver >= cur_ver then
    redis.call('HSET', KEYS[1], 'remain', db_remain, 'version', db_ver)
    return 'FIXED'
end
-- version 落后但 remain 不一致 → 强制修 remain，version 取较大值
if redis.call('HGET', KEYS[1], 'remain') ~= db_remain then
    local fix_ver = math.max(db_ver, cur_ver)
    redis.call('HSET', KEYS[1], 'remain', db_remain, 'version', fix_ver)
    return 'FORCE_FIXED'
end
return 'SKIP'

10.8 告警规则

条件动作
单 SKU 偏差 ≤ 阈值，自动修复成功仅记 reconciliation_log
偏差 > 阈值（如库存差 > 10）即时告警
1 小时内同 SKU 自动修复 > 3 次告警（可能有在线逻辑 bug）
db_remain < 0紧急告警，不自动修 DB
修复 CAS 失败写 reconciliation_diff + 告警

10.9 与在线写路径的配合（避免打架）

措施说明
修复也用 Lua CAS与写路径相同逻辑，不用无条件 SET
近期写入跳过updated_at 在 5s 内 → 跳过本次，下轮再对
只修 Redis 不修 DB默认规则；DB 异常须人工审批
修复不写 DB对账任务只有读 DB + 写 Redis 权限

10.10 三层一致性闭环

在线层：幂等 + DB 事务 + Redis CAS + 失败 DEL/503
    ↓（小概率偏差）
离线层：增量对账 → 自动修 Redis → 差异告警
    ↓（活动结束）
审计层：库存 vs 订单流水交叉对账（人工兜底）

11缓存可能滞后的场景

前提：即使采用当前全套机制，Redis 仍可能在一定时间内落后于 MySQL。 这是「跨存储非原子写 + 命中零 DB」的固有代价。 防超卖不依赖缓存准确（靠 DB WHERE remain > 0）； 展示准确性才受下列滞后影响。

11.1 滞后场景汇总

# 场景 原因 滞后窗口 是否超卖 主要靠谁修复
1 DB commit → Redis CAS 前 DB 与 Redis 非原子，中间有空隙 通常 1–5ms 否 写路径 CAS 完成
2 CAS 被拒绝（慢写 / 过期 version） 慢写携带旧 version，CAS 正确拒绝以防覆盖新值；Redis 暂未跟上 DB 秒级~分钟级（无后续写、不 miss 时） 否 再读 DB 重 CAS / DEL / 对账
3 Redis 更新失败，DEL key → 回填前 失败处理 DEL 后，缓存为空直至 Singleflight 回填 5–50ms 否 Singleflight 回填
4 命中零 DB，长期脏读 key 存在、未过期、无 miss → 永远不回源 分钟~小时（至对账或逻辑过期） 否 增量对账（5min）/ 活动对账（30s）
5 Caffeine 本地缓存滞后 多节点各自缓存；Pub/Sub 延迟或丢失 10ms~1s（= 本地 TTL） 否 写后失效 + Pub/Sub；库存类禁用 Caffeine
6 逻辑过期返回旧值 非库存类故意返回 stale 换性能（设计取舍） 秒级 否 异步刷新；库存类已排除
7 分段 total 与 segment 不同步 一次写涉及多 Redis key，非事务 分钟级 否 total 失败重试；对账校验 SUM
8 对账 SKIP（高写入期） updated_at 5s 内跳过，避免覆盖在线写 秒杀高峰持续数分钟 否 在线 CAS + CAS 失败告警
9 读回填误走从库（配置错误） miss 回填读到主从延迟前的旧值 = 主从延迟（10ms~1s+） 展示可能错 回填强制主库

11.2 典型场景详解

场景 1：写路径时间窗

T1  MySQL commit 成功（remain=30, version=10）
T2  读请求命中 Redis（仍是 remain=35, version=9）  ← 短暂脏读
T3  Lua CAS 更新 Redis 成功

场景 2：CAS 被拒绝（过期写 / stale write）

T1  写请求A：commit DB version=10，CAS 延迟
T2  写请求B：commit DB version=11，CAS Redis 成功
T3  写请求A：CAS 被拒（10 < 11）→ Redis 可能仍为旧值

说明：这是 version CAS 的正确防护（防止旧写覆盖新写），
      不是方案失效；拒绝后须补偿（再 CAS / DEL / 对账）。

场景 4：命中零 DB（影响最大）

Redis 中 remain 错误，但 key 存在、未逻辑过期
→ 所有读直接返回错误值，永远不回源
→ 最长滞后 = 逻辑过期 TTL 或下次对账周期

规避：库存类不走「过期返回旧值」；活动 SKU 30s 对账。

场景 5：多节点 Caffeine 不一致

节点1：写成功 → Redis 更新 → 本节点 Caffeine 失效
节点2：Caffeine 仍返回旧值（Pub/Sub 丢失）
→ 不同用户看到不同库存（节点间不一致）

规避：库存类禁用 Caffeine；或极短 TTL（100ms）+ Pub/Sub。

11.3 滞后窗口分级

层级典型滞后业务影响
写路径空隙毫秒级极短暂展示偏差，几乎无感
Caffeine 跨节点10ms ~ 1s不同用户看到不同数字
CAS 拒绝未补偿秒级 ~ 分钟级页面库存偏差
命中零 DB 脏读分钟级 ~ 小时级长期展示错误（靠对账收敛）
对账收敛5min（增量）/ 30s（活动）离线兜底上限

11.4 CAS 拒绝后补偿（与 §2 步骤 ⑥b 一致）

CAS 返回拒绝（0）时：
  ① 再读 DB 当前 (remain, version)
  ② 若 db.version > redis.version → 用 DB 最新值再 CAS 一次
  ③ 仍失败 → DEL key 或记告警，交给对账

11.5 方案其它局限

局限说明
架构复杂度高Lua CAS、version、分段、Caffeine、对账 — 小团队维护成本偏高
无法 Redis+MySQL 强一致不用 2PC/XA 前提下只能最终一致
写路径延迟同步 DB + Redis ≥ 5–10ms，高于纯异步 MQ 方案
热点 SKU 瓶颈分库分表不解决单行锁，仍须分段 + 削峰
Redis 弱依赖故障时写路径 503 / 读击穿 DB，非「可选缓存」
运维与监控要求高须监控 CAS 拒绝率、对账修复次数等
低 QPS 场景偏重数千 QPS 非秒杀场景可能过度设计

11.6 业务容忍度对照

业务要求本方案匹配度
10W TPS + 同步写 + 不超卖匹配
库存展示允许秒级误差匹配
写后列表立刻准确须绕缓存或接受延迟
跨节点展示强一致Caffeine 是短板
小团队、低 QPS偏重

12design-spec §4.6 落地摘要

项方案
真相源MySQL 分库分表
写入顺序幂等 → MySQL 事务提交 → Lua CAS 更新 Redis → 失效 Caffeine
读路径命中零 DB；写回 CAS；丢弃回填后再 GET Redis；库存类不同步返回旧值
缓存策略Cache-Aside + version CAS；常规不 DEL；Redis 失败时例外 DEL
击穿防护Singleflight；Caffeine 写后失效；库存读同步刷新
一致性DB 单分片强一致；缓存最终一致；跨存储可收敛；离线对账
离线对账见 §10：增量 5min + 活动 30s；以 DB 为准修 Redis
缓存滞后见 §11：9 类场景、窗口分级；防超卖靠 DB，展示准确靠对账
禁止项先写 Redis、延时双删、无条件 SET、Redis 失败静默降级、跨存储 2PC
